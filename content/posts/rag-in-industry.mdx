## 一个假想场景

想象车间里有一份**大几十页**的设备维护手册，工人遇到设备故障时的典型困境：

1. 翻纸质手册 → 找不到对应章节
2. PDF 关键词搜索 → 「风扇异响」搜不到，因为手册里写「散热单元噪音超标」
3. 喊师傅 → 师傅可能在另一个车间

**痛点：知识找不到、找不准、找得慢。**

如果用 RAG（检索增强生成）来解决，技术方案会长什么样？

## 方案思路

```text
工人提问 → 向量检索 → 匹配相关段落 → 大模型生成回答 → 返回原文出处
```

核心约束：**不让大模型"自由发挥"**，只让它基于检索到的真实段落作答。

## 技术选型思考

| 层 | 候选 | 选型逻辑 |
|---|---|---|
| 向量库 | Qdrant / Milvus / Chroma | 开源、性能、支持元数据过滤 |
| Embedding | bge-large-zh / multilingual-e5 | 中文场景，离线可部署 |
| LLM | 通义千问 / DeepSeek / Qwen 本地 | 国产模型，数据不出厂 |
| 框架 | LangChain / LlamaIndex | 生态成熟 |
| 前端 | Web / 小程序 / 微信公众号 | 看工人使用习惯 |

## 核心代码骨架（示例）

```python
from langchain.vectorstores import Qdrant
from langchain.embeddings import HuggingFaceEmbeddings
from langchain.text_splitter import RecursiveCharacterTextSplitter

# 1. 切片：保留章节标题作为元数据，便于追溯出处
splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=80,
    separators=["\n## ", "\n### ", "\n\n", "\n", "。"],
)

# 2. 向量化入库
embeddings = HuggingFaceEmbeddings(
    model_name="BAAI/bge-large-zh-v1.5",
    encode_kwargs={"normalize_embeddings": True},
)

vectordb = Qdrant.from_documents(
    documents=splitter.split_documents(docs),
    embedding=embeddings,
    url="http://qdrant:6333",
    collection_name="device_manual",
)

# 3. 检索 + 生成
def answer(question: str) -> dict:
    # 召回 Top-K 段落
    docs = vectordb.similarity_search(question, k=5)
    context = "\n\n".join(d.page_content for d in docs)

    prompt = f"""你是设备维护助手。仅根据下面的资料回答问题。
如果资料里没有答案，直接说"手册里没有相关内容，请联系工程师"。

【资料】
{context}

【问题】{question}
【回答】"""

    return {
        "answer": llm.invoke(prompt),
        "sources": [d.metadata for d in docs],  # 必须返回出处
    }
```

## 工业场景下的几个特殊考量

### 考量 1：通用 Embedding 对**专业术语**不敏感

「轴承异响」和「轴承噪音」在通用模型里语义距离可能很远，但实际意思接近。

**思路：** 用领域语料对 Embedding 模型做轻量微调（LoRA），或者维护一份**专业术语同义词表**做查询扩展。

### 考量 2：工人不会"提问"

工人可能说「那个东西不转了」，而不是「主轴电机失速」。

**思路：** 加一层 **「问题改写」** prompt，把口语化问题转成手册风格的查询词，再送进向量库检索。

### 考量 3：网络不稳定

车间 WiFi 信号可能差，云端 LLM 调用容易超时。

**思路：** 边缘服务器部署本地小模型（如 Qwen2.5-7B）做主力推理，云端只做兜底；同时所有数据**本地化**，避免出厂。

## 还在探索的方向

- **多模态 RAG**：让工人能拍张故障设备照片问"这是哪里坏了"
- **Agent 闭环**：让 AI 不只是答问，能调用工单系统帮工人开维护任务
- **效果评估**：用什么指标衡量「工人是否真用上了」

## 小结

> RAG 在工业场景的核心，不是模型多强，而是**领域适配做得多细**。

通用方案在工业场景往往效果一般，最后还是要在「术语处理 / 查询改写 / 本地化部署 / 工程稳定性」上下笨功夫。
