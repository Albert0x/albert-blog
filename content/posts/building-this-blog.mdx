## 为什么要做这个博客

我做工业互联网软件好几年了，平时也在自学 RAG、Agent 这类 AI 应用方向。但**一直没把这些零散的知识系统化沉淀下来**。

> 写博客最好的时间是十年前，其次是现在。

这次决定**不用任何现成模板**，从 `create-next-app` 一行命令开始，自己搭一个真正属于自己的博客。目标：

1. **不只是「能展示文章」**，要能展示**全栈工程能力**
2. **不只是「静态站」**，要有 AI 助手 / 点赞 / 评论 / 后台管理这些真实业务功能
3. **不能丑**，要能炫

最后这一周的成果，你现在看的就是。

## 三个我最满意的亮点

### 亮点 1：工业 + AI 的 RAG 博客助手

博客右下角那个 ✨ 浮动按钮 —— 点开就是一个**能聊我博客内容**的 AI 助手。

#### 它的工作流

```text
用户问题
   ↓
① 用 MiniSearch 在博客知识库（所有 MDX 文章切片）里检索 Top 3 相关片段
   ↓
② 把片段塞进 system prompt 作为上下文
   ↓
③ 流式调用 DeepSeek 生成回答
   ↓
④ 在结尾自然引用源文章 + 通过 message metadata 把文章 slug 返回前端
   ↓
⑤ 前端渲染回答（Markdown + 流式打字） + 引用文章卡片
```

#### 关键代码（chat API）

```ts
// app/api/chat/route.ts
const hits = searchKnowledge(query, 3);  // 检索

const systemPrompt = `你是博主的 AI 助手...
# 本次问题相关的文章片段（已检索）
${hits.map(h => `《${h.postTitle}》${h.content}`).join("\n\n---\n\n")}

# 回答规则
1. 严格基于上下文回答，不要编造
2. 如用到某篇文章，在结尾加「📚 相关文章：《xxx》」
...`;

const result = streamText({
  model: deepseek(DEFAULT_MODEL),
  system: systemPrompt,
  messages: modelMessages,
  maxOutputTokens: 800,
});

return result.toUIMessageStreamResponse({
  messageMetadata: ({ part }) => {
    if (part.type === "finish") {
      return { sources: hits.map(h => ({ slug: h.postSlug, title: h.postTitle })) };
    }
  },
});
```

#### 中文分词的小巧思

MiniSearch 默认按空格切，对中文不友好。我用了**英文按单词 + 中文 2-gram**的混合策略：

```ts
tokenize: (text) => {
  const tokens: string[] = [];
  // 英文 / 数字按单词切
  tokens.push(...(text.match(/[A-Za-z0-9]+/g) || []));
  // 中文只保留 CJK 汉字，按相邻两字一组
  const cn = text.replace(/[^一-鿿]+/g, "");
  for (let i = 0; i < cn.length - 1; i++) {
    tokens.push(cn.slice(i, i + 2));
  }
  // 单字也保留，提升召回
  tokens.push(...cn.split(""));
  return tokens;
},
```

> 工业级 RAG 会用向量检索（Embeddings + Qdrant）+ Re-ranking 重排，对个人博客这个量级，关键词 BM25 已经够用。

### 亮点 2：自建 GitHub OAuth 后台管理

最初我把博客写好后，发现**每次写文章都要在本地编辑 + git push** —— 在手机 / 平板 / 同事电脑上根本写不了。

于是我加了一套**自建后台**：

#### 流程

```
打开 /admin → 没登录? → 跳 /login
   ↓ 点「GitHub 登录」
GitHub OAuth 授权 → 拿到 access_token
   ↓
NextAuth 拦截 signIn callback → 白名单校验（只有我能登）
   ↓ ✅
进入 /admin → 看到文章列表（实时从 GitHub API 拉）
   ↓ 点「编辑」
Monaco Editor 加载 MDX 内容 + frontmatter 表单
   ↓ 点「更新发布」
后端调 GitHub API 用 access_token 提交 commit
   ↓
Vercel 检测到 push 自动重新部署
   ↓
1-2 分钟后线上更新
```

#### 关键代码（GitHub API 写文件）

```ts
// lib/github.ts
export async function upsertFile(
  token: string, path: string, content: string,
  message: string, sha?: string
) {
  const octo = new Octokit({ auth: token });
  return octo.repos.createOrUpdateFileContents({
    owner: "Albert0x", repo: "albert-blog", branch: "main",
    path, message,
    content: Buffer.from(content, "utf8").toString("base64"),
    sha,  // 更新现有文件必传 sha（GitHub 的乐观锁）
  });
}
```

整套后台**没有任何数据库**——文章是 Git 仓库的 MDX 文件，用户身份是 GitHub OAuth，发布触发 Vercel 部署。**0 维护成本**。

### 亮点 3：0 数据库的匿名点赞系统

文章底部那个 ❤️ 点赞按钮，**任何访客都能点**（不登录），**Cookie 永久去重**。

#### 数据存哪？

**Upstash Redis**（Vercel Marketplace 直接集成，免费 10000 commands/月）。

#### 关键代码

```ts
// app/api/likes/[slug]/route.ts
// 点赞接口三道防线：
// 1) Origin 校验（只让本站调用）
// 2) IP 速率限制（防爆破）
// 3) Cookie 去重（每浏览器每篇 1 次）

if (cookieStore.has(VOTED_COOKIE(slug))) {
  return Response.json({ count: await getLikes(slug), alreadyVoted: true });
}

const newCount = await incrementLikes(slug);  // Redis INCR 原子加
cookieStore.set(VOTED_COOKIE(slug), "1", {
  maxAge: 60 * 60 * 24 * 365,  // 1 年
  httpOnly: true, sameSite: "lax", secure: true,
});
return Response.json({ count: newCount, voted: true });
```

点击瞬间数字 +1（**乐观更新**），同时弹出 3 个**带随机偏移的飘心粒子**：

## 我踩过的 10 个坑（按吃亏程度排序）

### 1. 🔴 Vercel 部署到 `*.vercel.app` 在大陆访问不稳定

**症状**：朋友说打不开博客
**原因**：`*.vercel.app` 域名段在部分运营商被干扰
**对策**：
- 短期：接受现实 + 重要场合发 X / 知乎链接（带博客地址）
- 中期：买域名 + Cloudflare DNS 代理（CF 国内节点不快但能开）
- 长期：买域名 + ICP 备案 + 国内 CDN

### 2. 🔴 EdgeOne Pages 默认排除中国大陆访问

**症状**：部署到 EdgeOne 后大陆访问 401 Unauthorized
**原因**：腾讯云 EdgeOne 对未备案项目**默认排除大陆**
**对策**：要么备案，要么放弃 EdgeOne 这条路

### 3. 🟠 NextAuth v5 + middleware 导致 `/admin/login` 死循环

**症状**：`ERR_TOO_MANY_REDIRECTS`
**原因**：`app/admin/login/page.tsx` 也被 `app/admin/layout.tsx` 守卫了，layout 未登录就 redirect 到 login，login 又走 layout
**对策**：把登录页从 `/admin/login` 移到顶级 `/login`，绕开守卫 layout

### 4. 🟠 Server Component 不能传 `onClick` 函数

**症状**：`Error: Event handlers cannot be passed to Client Component props`
**原因**：RSC 渲染时把 vDOM 序列化成 JSON，函数没法序列化
**对策**：重构 DOM 结构消除 onClick 需求（拆开嵌套链接），或者标 `"use client"`

### 5. 🟠 AI SDK v6 `convertToModelMessages` 在 DeepSeek 下报 `messages.some is not a function`

**症状**：streamText 内部 `standardizePrompt` 崩
**原因**：`@ai-sdk/openai@3.x` 默认走 OpenAI 新版 `/v1/responses` 接口，**DeepSeek 不支持**（只支持经典 `/v1/chat/completions`）
**对策**：换 `@ai-sdk/openai-compatible` 适配器 + 手动转换 UIMessage 到 ModelMessage

```ts
// 之前（炸）
import { createOpenAI } from "@ai-sdk/openai";
// 现在（稳）
import { createOpenAICompatible } from "@ai-sdk/openai-compatible";
const deepseek = createOpenAICompatible({
  baseURL: "https://api.deepseek.com/v1",
  apiKey: process.env.DEEPSEEK_API_KEY,
});
```

### 6. 🟠 Giscus `messageMetadata` 写法错误导致客户端崩溃

**症状**：评论卡片重复显示 + 控制台报错
**原因**：`messageMetadata: () => ({ sources })` 这种写法**每个 stream part 都会调一次**，前端 messages 数组结构错乱
**对策**：按 `part.type === "finish"` 判断只在流结束时附加一次

### 7. 🟡 TypeScript target 太老不支持 `\p{P}` Unicode 属性

**症状**：build 报 `This regular expression flag is only available when targeting 'es6' or later`
**对策**：要么升级 tsconfig target 到 ES2018+，要么用 `[^一-鿿]` 这种 unicode 范围反向匹配（兼容 ES5+）

### 8. 🟡 Map.entries() 迭代要 ES2015+ target

**症状**：`Type 'MapIterator<...>' can only be iterated through when using --downlevelIteration or --target of 'es2015'`
**对策**：tsconfig.json 加 `"target": "ES2020"`

### 9. 🟡 `@vercel/kv` 已被 deprecated

**症状**：装包时 pnpm 提示弃用
**对策**：改用 `@upstash/redis`（Vercel KV 的底层）

### 10. 🟡 GitHub commit 暴露真实姓名

**症状**：GitHub 仓库每个 commit 显示我的真名拼音 + 工作邮箱
**原因**：git 全局 `user.name` 和 `user.email` 是多年前配的工作信息
**对策**：用 git `includeIf` 指令**按目录自动切换身份**

```ini
# ~/.gitconfig
[user]
    name = 真名（工作仓库默认用这个）
    email = work@xxx.com

[includeIf "gitdir:D:/Code/personal/"]
    path = ~/.gitconfig-personal
```

```ini
# ~/.gitconfig-personal
[user]
    name = Albert
    email = 12345678+Albert0x@users.noreply.github.com
```

这样工作仓库自动用工作号，个人项目自动用 noreply 邮箱保护身份。
