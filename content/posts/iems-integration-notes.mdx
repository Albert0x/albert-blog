## 背景设定（虚构场景）

假设我们要给一家中等规模的制造企业，搭建一套**能源管理系统**：监控车间用电、用水、用气，并按工艺段做能耗分析。如果让我来设计，整体架构会怎么走？

## 整体架构思路

```text
[现场设备] → [工业网关] → [能源管理平台] → [API 网关] → [Web 看板]
   电表/水表     协议转换       数据采集存储      权限/聚合      可视化
```

关键设计点：**平台本身不要直接对外暴露**，前面架一层自己的 API 网关，做权限、缓存、聚合。

## 思考点 1：网关协议混乱怎么办

现场设备协议可能五花八门：
- 电表常见 Modbus RTU（串口）
- PLC 常见 Modbus TCP
- 新设备可能直接支持 MQTT

**思路：** 统一在网关做协议转换，对上层只暴露一种协议（比如 Modbus TCP），让上游应用简单。

```python
# 老王说明：示例代码 - 从能源平台拉数据的封装
import httpx
from typing import Any

class EnergyPlatformClient:
    """能源平台 REST API 客户端示例"""

    def __init__(self, base_url: str, token: str):
        self.base_url = base_url.rstrip("/")
        self.headers = {"Authorization": f"Bearer {token}"}

    async def get_realtime(self, device_id: str) -> dict[str, Any]:
        """拉取设备实时数据点位"""
        url = f"{self.base_url}/api/v2/devices/{device_id}/realtime"
        async with httpx.AsyncClient(timeout=5.0) as client:
            resp = await client.get(url, headers=self.headers)
            resp.raise_for_status()
            return resp.json()
```

## 思考点 2:时序数据库选型

能源数据是典型的**高频写入 + 时序查询**场景。常见思路：

| 数据频率 | 存储 | 用途 |
|---------|------|------|
| 秒级 | InfluxDB / TDengine | 实时监控大屏 |
| 分钟级 | InfluxDB / PostgreSQL | 报表分析 |
| 小时级 | PostgreSQL | 长期归档 |

不同频率分层存储，是工业时序数据的常规做法。

## 思考点 3：权限模型设计

工业系统常见的权限需求：
- **角色权限**：管理员 / 操作工 / 查看者
- **数据权限**：按车间 / 工艺段 / 设备组隔离

如果平台本身只提供「角色 → 菜单」的二维结构，**数据维度的权限**就需要在自建 API 层补一层过滤。

## 还在思考的方向

- **报警风暴抑制**：同一设备短时间多次告警如何聚合
- **断网重连数据补偿**：网关离线期间数据如何回补
- **多租户隔离**：一个平台支撑多家工厂时的数据隔离

## 小结

工业系统的灵魂是 **「稳定 > 性能 > 美观」**。

很多通用平台不是银弹，它给你一个起点，但差异化需求还是要靠自己撸代码补齐。这也是为什么工业开发者需要既懂业务、又懂工程。
